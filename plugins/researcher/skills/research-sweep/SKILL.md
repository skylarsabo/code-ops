# RESEARCH SWEEP — Run the Whole Research Pipeline End-to-End

**Invoked as `/researcher:research-sweep`.** First read the `${CLAUDE_PLUGIN_ROOT}/CONVENTIONS.md` bundled with this plugin (search the plugin directory for it if needed) — it defines the research integrity & egress model (§A), the operating model, the interaction protocol, modes (§5), the finding/idea schema and tiers (§6, §7), severity & priority (§8), the quality lenses (§10), the hand-off map (§11), and the shared-artifact / SSOT rules (§12). This skill **orchestrates the researcher's own skills in sequence** as one developer-in-the-loop pipeline — it does not replace them. It runs them in a sensible order, carries the shared registers forward fresh, keeps a master plan, and pauses at every phase boundary, **always surfacing `EGRESS_MANIFEST.md` at each checkpoint**.
**Mode:** orchestrator (DISCOVERY + REVIEW; never DOCUMENT-only) · **Produces:** the consolidated `RESEARCH_FINDINGS.md`, `IDEAS_REGISTER.md`, per-spike design briefs, `EGRESS_MANIFEST.md`, and a single `EXECUTIVE_SUMMARY.md`. Never edits code (§A, §4).

**Requires nothing beyond the researcher plugin.** It composes `code-ops-suite` / `rigor` / `privacy-opsec-suite` skills only when installed for hand-off (§11); the documentation lookup is local-first via lib-docs (§2). Web research is opt-in and runs only through the composed deep-research skill behind a checkpoint (§A).

## Phase 0 — Scope the run & egress permission  *(checkpoint)*
Detect the stack, the repo size, and the question(s) driving this sweep; record the HEAD `<sha>` everything will be verified against (§12). Then confirm with me:
- **Which modes run, and in what order** — any of: **research-improve** (grounded improvements) · **research-ideate** (net-new directions) · **library-eval** (an adopt-X decision) · **research-spike** (a design brief for a named task) · **ecosystem-watch** (what changed in our stack to act on). Default: ground → improve + ideate → verify → propose. **research-verify** is not optional — it gates every load-bearing claim before publish (§7).
- **Scope** (areas/questions in or out), appetite (quick wins vs. big bets), and the **hand-off targets** you care about (§11).
- **Egress permission and scope.** State plainly: by default this sweep is local-first — codebase, version-control history, installed-dependency docs (via lib-docs, §2), and materials you hand me. **Web/external retrieval is opt-in and off unless you grant it here.** If you grant it, name the allowed scope (which questions may go to the web, which hosts are acceptable); every external request will be recorded in `EGRESS_MANIFEST.md` and surfaced to you, and no artifact will be published citing an un-manifested source (§A).

Open a master todo and a running `EXECUTIVE_SUMMARY.md` that spans phases. Initialize `EGRESS_MANIFEST.md` for the dated run folder (§12). **Surface any critical finding — a live secret, a known-exploited CVE, a broken trust boundary — to me immediately, in any phase, as a hand-off (§A, §4).**

> **CHECKPOINT (before any network egress):** present the planned phase order, the scope, and the egress decision in plain terms — what (if anything) is permitted to leave the machine, why, and that every request will be disclosed in `EGRESS_MANIFEST.md`. **No web request happens before you approve this.** Proceed within the agreed plan; keep local gathering moving while a decision is pending (§3).

## Phase 1 — Ground in our code (local, zero egress)
Establish the grounding baseline every later phase ties back to (§A grounding rule). Map the relevant architecture, the constraints, and the prior art already present in-repo; verify library/framework facts against the **installed versions** through the local docs lookup (lib-docs, or the `code-ops-docs` MCP when `code-ops-suite` is installed) — never from memory (§2, §10 source-quality). Pull version-control history for *why* the code is the way it is. This phase causes no egress. → *Checkpoint:* the grounded picture (what holds for our code, given our constraints) and the open questions the next phase must answer.

## Phase 2 — Gather  *(checkpoint — surfaces the manifest)*
Run the selected discovery skills against the grounded baseline, fanning out read-only (§1): **research-improve** for grounded improvement candidates, **research-ideate** for net-new directions, **library-eval** for an adopt/replace/build decision, **research-spike** for a named-task design brief, **ecosystem-watch** for dependency updates / CVEs / deprecations / newly-available capabilities. Each emits entries on the finding/idea schema with sources, tier, grounding, and a disconfirmation pass (§6, §7); collect them into `RESEARCH_FINDINGS.md` and `IDEAS_REGISTER.md`, each stamped `Verified-at: <sha>` (§12). **If web egress was granted in Phase 0**, the gathering skills compose the deep-research skill within the approved scope, and **every external request is recorded** the moment it happens: `node ${CLAUDE_PLUGIN_ROOT}/scripts/research-manifest.mjs record --tool <tool> --host <host> --url <url> --why "<reason>"` (§A). Prefer the installed-version docs and primary sources before reaching outward (§7 triangulate). → *Checkpoint:* the raw candidate registers, the gaps still open, **and the current `EGRESS_MANIFEST.md`** so you can see exactly what (if anything) left the machine.

## Phase 3 — Verify  *(checkpoint)*
Run **research-verify** over every *load-bearing* claim, recommendation, and external "fact" the gathering phase produced — adversarially, against both our code and the cited sources (§7, §10 grounding). Re-tier on the evidence: confirm against our code or a primary source → CONFIRMED; ≥2 independent sources or one strong primary → PROBABLE; a single weak/secondary lead → SPECULATIVE; anything unsupported or un-grounded is dropped or marked `UNVERIFIED` and never presented as fact (§A, §7). Apply the disconfirmation pass once more before carry-forward — already done here? incompatible with our stack? superseded? measured, or merely assumed? (§A). Re-validate the registers mechanically before they cross this boundary: `node ${CLAUDE_PLUGIN_ROOT}/scripts/revalidate-register.mjs <register> --root <repo>`, dropping or `OBSOLETE-AT <sha>`-stamping anything that no longer holds (§12). → *Checkpoint:* the verified, re-tiered registers — CONFIRMED items separated from PROBABLE/SPECULATIVE — and any claim that failed verification, **with the manifest still in view**.

## Phase 4 — Propose & hand off  *(checkpoint)*
Rank the survivors by **value × reach ÷ effort**, weighted by tier and grounding strength (§8); for each, give the concrete recommendation, the smallest valuable slice, the trade-offs, and the **hand-off target** — never "consider maybe" (§9). Map each actionable item to its implementer (§11): grounded improvements → `code-ops-suite:remediation` / `rigor:fix-verified`; features/ideas → `code-ops-suite:feature-discovery` / `code-ops-suite:feature-implementation` / `code-ops-suite:ship`; an adopt/replace decision or a load-bearing design choice → `code-ops-suite:adr`; measured performance work → `rigor:improve-measured`; dependency/CVE actions → `code-ops-suite:dependency-upgrade` / `privacy-opsec-suite:supply-chain-trust`. Each design brief meets the documentation quality standard (§13) and is concrete enough that its implementer can act without re-researching (§11). The researcher proposes and hands off — **it never edits code** (§A, §4). → *Checkpoint:* the ranked, hand-off-ready registers and briefs.

## Phase 5 — Consolidate the executive summary  *(final, validate before publish)*
Tie the whole sweep together into one `EXECUTIVE_SUMMARY.md`: the question(s) asked, the grounded answer, and the unified picture across findings, ideas, and design briefs — **CONFIRMED items separated from PROBABLE/SPECULATIVE** (§7), led by the highest-value, best-grounded item, each carrying its tier, its sources, and its hand-off target (§8, §11). Run a final self-audit for generic filler, fabricated sources/capabilities, and any un-grounded claim presented as fact (§4, §10). **Before publishing any artifact**, validate the egress discipline fail-closed: `node ${CLAUDE_PLUGIN_ROOT}/scripts/research-manifest.mjs validate <artifact>` for each register, brief, and the summary — a published artifact must not cite a web source absent from `EGRESS_MANIFEST.md`, and the script blocks (non-zero exit) if it does (§A, §12). All artifacts live in the dated run folder (§12).

## Done when
Every selected phase is complete, its deliverable produced, and its checkpoint passed; the egress decision was made explicitly at Phase 0 and **no web request occurred without it**; every external request is recorded in `EGRESS_MANIFEST.md` and was surfaced at the checkpoints; every load-bearing claim was run through **research-verify**, cited, and tiered (§7), with CONFIRMED separated from PROBABLE/SPECULATIVE; the carried-forward registers are fresh (no obsolete item re-shown, each stamped `Verified-at: <sha>`, §12); every actionable item maps to a real implementer skill (§11); `research-manifest.mjs validate` passed fail-closed on each published artifact (§A); no code was changed (§4); and `EXECUTIVE_SUMMARY.md` ties findings, ideas, and briefs into one consistent picture. Present the summary first — highest-value, best-grounded item at the top — then list anything still awaiting your decision.
