# ECOSYSTEM WATCH — What Changed In Our Stack, And What To Do About It

**Invoked as `/researcher:ecosystem-watch`.** First read the `${CLAUDE_PLUGIN_ROOT}/CONVENTIONS.md` bundled with this plugin (search the plugin directory for it if needed) — it defines the research integrity & egress model (§A), the operating model, interaction protocol, safety rails, schemas, evidence tiers, and quality lenses this skill references by section.
**Mode:** DISCOVERY · **Produces:** `ECOSYSTEM_WATCH.md` (a tiered, cited register of actionable changes), `EGRESS_MANIFEST.md`, `EXECUTIVE_SUMMARY.md`.

Surface what moved underneath us — new releases, security advisories, deprecations, and newly-available capabilities in the dependencies and platforms **we actually use** — and turn it into a small, ranked, grounded backlog of things worth acting on. Every item is tiered and cited (§7), disconfirmed against our tree (§A), and handed off to an implementer (§11). This skill researches and proposes; it **never edits source**. It is a good fit for a recurring schedule: run it, diff against last run's register, and act only on what is new and relevant.

## Phase 0 — Inventory our stack, set scope, decide on egress  *(checkpoint)*
Detect the ecosystem(s): package manager(s) and manifests/lockfiles, language/runtime version pins, container/base images, CI toolchain, and any platform SDKs. Read the **prior `ECOSYSTEM_WATCH.md`** if one exists so this run is a diff, not a restart. Frame the watch scope: which dependency tiers (direct only vs. transitive), which change classes (security only, or also deprecations + new capabilities), and the freshness window since the last run. Note that change-gathering generally requires **opt-in web egress** (advisory feeds, release notes) — that is the central decision below.
> **CHECKPOINT:** present the detected stack, the proposed scope (tiers · change classes · since-when), and an explicit **egress plan** — which hosts/feeds you would query and why. Get the developer's opt-in for network egress, or confirm a **local-only** run (lockfile + installed-doc grounding only). No request leaves the machine before this is approved (§A, §3). If approved, the egress budget agreed here bounds Phase 2.

## Phase 1 — Ground: what we actually use, at what version  *(local-first, no egress)*
Build the authoritative "what we run" picture from local sources only. Resolve **installed/locked versions** from the lockfile (not the loose ranges in the manifest), and confirm each dependency is genuinely **in use** by tracing imports/usage in source (`file:line`) — an outdated package nothing imports is a removal candidate, not an upgrade. Read installed-dependency docs and types via `${CLAUDE_PLUGIN_ROOT}/scripts/lib-docs.mjs` (or the `code-ops-docs` MCP `get-docs` if `code-ops-suite` is installed) to capture each dep's **current, installed** surface and its own stated support/deprecation notices — local, zero query-egress (§2). Record runtime/base-image pins and EOL-relevant versions. This grounded baseline is what every Phase 2 change is checked against, so a change to a version we don't run, or an API we don't call, is filtered out before it ever becomes a finding.

## Phase 2 — Gather changes  *(opt-in web; every request recorded)*
Only with the Phase 0 opt-in, and only within the agreed hosts and budget. Compose the `deep-research` skill for the actual fetch-and-triangulate work; for each dependency/platform in the grounded baseline, gather: **new releases** (changelogs, release notes), **security advisories / CVEs** (with severity and affected version ranges), **deprecations and EOL dates**, and **newly-available capabilities** that we currently work around or build by hand. Prefer primary sources — the project's own release notes, the advisory database entry, the maintainer's deprecation notice — over secondary write-ups (§7, §10 source-quality).
- **Record every external request** as it happens via `node ${CLAUDE_PLUGIN_ROOT}/scripts/research-manifest.mjs record --tool <tool> --host <host> --url <url> --why "<reason>"` (time · tool · host · url · why). A published item must not cite a source absent from the manifest — the validator fails closed on this (§A, §12).
- Redact any secret/PII encountered to `<REDACTED:reason>`; never paste credentials into the manifest or a finding (§4).
- Stay inside the budget. If the watch needs more reach than was approved, **stop and return to a checkpoint** rather than widening egress silently.

## Phase 3 — Triage & disconfirm: does it affect us?  *(compose `research-verify`)*
For each gathered change, run the disconfirmation pass against the grounded baseline (§A) before it earns a register entry. Ask, with evidence: does the **affected version range** include the version we actually run? Do we **call the affected/deprecated API** (cite the `file:line`, or confirm we don't and drop it)? Is it already mitigated, pinned, or behind a flag here? Is a CVE reachable given how we use the dep, or theoretical for our usage? Compose `research-verify` to adversarially confirm the high-impact and security-class claims against both the source and our code, and to tier the verdict — **CONFIRMED** (affected version + reachable usage, verified against our tree or a primary advisory) · **PROBABLE** · **SPECULATIVE** (§7). Discard generic ecosystem noise that doesn't touch us; that filtering is the value of this skill.

## Phase 4 — Register actionable items & hand off  *(checkpoint)*
Write surviving items to `ECOSYSTEM_WATCH.md` using the finding schema (§6): `RSCH-NNN` · title · lens · tier · claim · sources (lockfile entry · installed-doc · external + manifest line) · `Verified-at: <sha>` · grounding (the `file:line` we run that it touches) · disconfirmation (what you ruled out) · value/impact · smallest slice · recommendation · **hands-off-to** · effort · risks. Rank by value × reach ÷ effort, weighted by tier and grounding (§8); lead with the top items, security first. Map each to its implementer (§11): security CVEs and routine/major upgrades → `code-ops-suite:dependency-upgrade`; egress/telemetry/provenance/integrity concerns introduced or revealed by a change → `privacy-opsec-suite:supply-chain-trust`; a newly-available capability worth adopting → `feature-discovery` / `feature-implementation`; an architectural choice it forces (e.g. an EOL runtime migration) → `adr`. Before publishing, re-confirm each entry still holds with `node ${CLAUDE_PLUGIN_ROOT}/scripts/revalidate-register.mjs ECOSYSTEM_WATCH.md --root <repo>`, and validate the artifact's citations with `node ${CLAUDE_PLUGIN_ROOT}/scripts/research-manifest.mjs validate ECOSYSTEM_WATCH.md` (fails closed on any un-manifested external claim, §12).
> **CHECKPOINT:** present the ranked register (security-class items and reachable CVEs first) and the egress manifest for this run. The developer decides what to hand off now, what to schedule, and what to accept/defer with rationale. The researcher hands off; it does not perform the upgrades.

## Recurring schedule
This watch is designed to run on a cadence. Each scheduled run is a **diff against the last `ECOSYSTEM_WATCH.md`**: re-ground (Phase 1), gather only changes since the prior run's `Verified-at` sha within the standing egress opt-in and budget, drop entries that no longer apply (stamp `OBSOLETE-AT <sha>`, §12), and surface only what is new and reachable. A scheduled run still honors §A — it operates within a pre-agreed egress scope and records every request in the manifest; if a run would exceed that scope, it stops at a checkpoint rather than widening egress unattended.

## Done when
The grounded baseline reflects installed/locked versions and confirmed in-use deps; every gathered change is triaged against our tree and either registered or dropped with a reason; no claim is un-cited and no external claim is un-manifested (`research-manifest.mjs validate` passes); each register entry is tiered, stamped `Verified-at: <sha>`, and mapped to an implementer (§11); the register is re-validated (`revalidate-register.mjs`), ranked security-first, and deduped against the prior run; the egress manifest is surfaced and within the approved budget; both checkpoints are done; no code changed. Reference `CONVENTIONS.md` throughout. Present `EXECUTIVE_SUMMARY.md` first, the highest-severity reachable change at the top.
