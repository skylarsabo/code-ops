# RESEARCH SWEEP — Run the Whole Research Pipeline End-to-End

**Codex path rule:** Resolve `<plugin-root>` as the installed root of this plugin (the directory containing `CONVENTIONS.md`); use it for every bundled script or reference path.

**Invoke in Codex by naming `researcher:research-sweep`.** First read the bundled `<plugin-root>/CONVENTIONS.md` — the research integrity & egress model (§A), modes (§5), schema/tiers (§6, §7), priority (§8), lenses (§10), hand-off map (§11), SSOT rules (§12). This skill **orchestrates the researcher's own skills in sequence** as one developer-in-the-loop pipeline — it does not replace them. It carries the shared registers forward fresh, keeps a master plan, pauses at every phase boundary, and **always surfaces `EGRESS_MANIFEST.md` at each checkpoint**.
**Mode:** orchestrator (DISCOVERY + REVIEW; never DOCUMENT-only) · **Produces:** consolidated `RESEARCH_FINDINGS.md`, `IDEAS_REGISTER.md`, per-spike design briefs, `EGRESS_MANIFEST.md`, one `EXECUTIVE_SUMMARY.md`. Never edits code (§A, §4).

**Requires nothing beyond the researcher plugin.** It composes `code-ops-suite` / `rigor` / `privacy-opsec-suite` skills only when installed, for hand-off (§11).

## Phase 0 — Scope the run & egress permission  *(checkpoint)*
Detect the stack, repo size, and the question(s) driving the sweep; record the HEAD `<sha>` everything is verified against (§12). Confirm with me:
- **Which modes run, in what order** — any of **research-improve** · **research-ideate** · **library-eval** · **research-spike** · **ecosystem-watch**. Default: ground → improve + ideate → verify → propose. **research-verify** is not optional — it gates every load-bearing claim before publish (§7).
- **Scope** (areas/questions in or out), appetite (quick wins vs. big bets), and the **hand-off targets** you care about (§11).
- **Egress permission and scope.** Default is local-first — codebase, VCS history, installed-dependency docs (lib-docs, §2), and materials you hand me. **Web/external retrieval is opt-in and off unless granted here.** If granted, name the allowed scope (which questions, which hosts); every external request is recorded in `EGRESS_MANIFEST.md` and surfaced, and no artifact is published citing an un-manifested source (§A).

Open a master todo and a running `EXECUTIVE_SUMMARY.md`; initialize `EGRESS_MANIFEST.md` in the dated run folder (§12). **Surface any critical finding — a live secret, a known-exploited CVE, a broken trust boundary — to me immediately, in any phase, as a hand-off (§A, §4).**
> **CHECKPOINT (before any network egress):** present the planned phase order, the scope, and the egress decision in plain terms — what (if anything) may leave the machine, why, and that every request will be disclosed in `EGRESS_MANIFEST.md`. **No web request happens before you approve.** Keep local gathering moving while a decision is pending (§3).

## Phase 1 — Ground in our code  *(local, zero egress)*
Establish the grounding baseline every later phase ties back to (§A grounding rule): map the relevant architecture, constraints, and in-repo prior art; verify library/framework facts against the **installed versions** (lib-docs, or the `code-ops-docs` MCP when `code-ops-suite` is installed) — never from memory (§2, §10 source-quality); pull VCS history for *why* the code is the way it is. → *Checkpoint:* the grounded picture and the open questions the next phase must answer.

## Phase 2 — Gather  *(checkpoint — surfaces the manifest)*
Run the selected discovery skills against the grounded baseline, fanning out read-only (§1): **research-improve**, **research-ideate**, **library-eval**, **research-spike**, **ecosystem-watch**. Each emits entries on the finding/idea schema with sources, tier, grounding, and a disconfirmation pass (§6, §7); collect into `RESEARCH_FINDINGS.md` and `IDEAS_REGISTER.md`, each stamped `Verified-at: <sha>` (§12). **If web egress was granted**, the gathering skills compose the deep-research skill within the approved scope, and **every external request is recorded the moment it happens**: `node <plugin-root>/scripts/research-manifest.mjs record --tool <tool> --host <host> --url <url> --why "<reason>"` (§A). Prefer installed-version docs and primary sources (§7 triangulate). → *Checkpoint:* the raw candidate registers, the gaps still open, **and the current `EGRESS_MANIFEST.md`** — exactly what (if anything) left the machine.

## Phase 3 — Verify  *(checkpoint)*
Run **research-verify** over every *load-bearing* claim, recommendation, and external "fact" — adversarially, against both our code and the cited sources (§7, §10 grounding). Re-tier on the evidence per the §7 tiers (CONFIRMED / PROBABLE / SPECULATIVE); anything unsupported or un-grounded is dropped or marked `UNVERIFIED`, never presented as fact (§A, §7). Apply the disconfirmation pass once more before carry-forward — already done here? incompatible with our stack? superseded? measured, or merely assumed? (§A). Re-validate the registers mechanically before they cross this boundary: `node <plugin-root>/scripts/revalidate-register.mjs <register> --root <repo>`, dropping or `OBSOLETE-AT <sha>`-stamping anything that no longer holds (§12). → *Checkpoint:* the verified, re-tiered registers — CONFIRMED separated from PROBABLE/SPECULATIVE — any claim that failed, **with the manifest still in view**.

## Phase 4 — Propose & hand off  *(checkpoint)*
Rank survivors by **value × reach ÷ effort**, weighted by tier and grounding (§8); for each, the concrete recommendation, the smallest valuable slice, the trade-offs, and the **hand-off target** — never "consider maybe" (§9). Map to implementers (§11): grounded improvements → `code-ops-suite:remediation` / `rigor:fix-verified`; features/ideas → `code-ops-suite:feature-discovery` / `code-ops-suite:feature-implementation` / `code-ops-suite:ship`; an adopt/replace or load-bearing design decision → `code-ops-suite:adr`; measured performance work → `rigor:improve-measured`; dependency/CVE actions → `code-ops-suite:dependency-upgrade` / `privacy-opsec-suite:supply-chain-trust`. Each design brief meets §13 and is concrete enough that its implementer can act without re-researching (§11). **The researcher never edits code** (§A, §4). → *Checkpoint:* the ranked, hand-off-ready registers and briefs.

## Phase 5 — Consolidate the executive summary  *(final, validate before publish)*
One `EXECUTIVE_SUMMARY.md`: the question(s), the grounded answer, the unified picture across findings, ideas, and briefs — **CONFIRMED separated from PROBABLE/SPECULATIVE** (§7), led by the highest-value best-grounded item, each with its tier, sources, and hand-off target (§8, §11). Final self-audit for generic filler, fabricated sources/capabilities, and un-grounded claims presented as fact (§4, §10). **Before publishing any artifact**, validate egress fail-closed: `node <plugin-root>/scripts/research-manifest.mjs validate <artifact>` for each register, brief, and the summary — a published artifact must not cite a web source absent from `EGRESS_MANIFEST.md`; the script blocks (non-zero exit) if it does (§A, §12). All artifacts live in the dated run folder (§12).

## Done when
Every selected phase complete, its deliverable produced, its checkpoint passed; the egress decision was made explicitly at Phase 0 and **no web request occurred without it**; every external request recorded in `EGRESS_MANIFEST.md` and surfaced at the checkpoints; every load-bearing claim ran through **research-verify**, cited, and tiered (§7), CONFIRMED separated from PROBABLE/SPECULATIVE; the carried-forward registers fresh (no obsolete item re-shown, each stamped `Verified-at: <sha>`, §12); every actionable item mapped to a real implementer skill (§11); `research-manifest.mjs validate` passed fail-closed on each published artifact (§A); no code changed (§4); `EXECUTIVE_SUMMARY.md` ties findings, ideas, and briefs into one consistent picture. Present the summary first — highest-value, best-grounded item at the top — then anything still awaiting your decision.
